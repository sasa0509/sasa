SSM框架整合

一、框架知识点回顾

1.1、MyBatis核心知识点

​ MyBatis 是一款优秀的==持久层框架==，它对**JDBC**操作数据库的过程进行封装，使开发者只需要关注 **SQL本身，而不需要花费精力去处理例如注册驱动、创建Connection、创建Statement、手动设置参数、结果集检索等JDBC**繁杂的过程代码。

**MyBatis应用程序根据XML配置文件创建SqlSessionFactory，SqlSessionFactory在根据Mapper映射配置文件或是DAO层代码的注解，获取一个SqlSession。SqlSession包含了执行SQL所需要的所有方法，可以通过SqlSession实例直接运行映射的SQL语句，完成对数据的增删改查和事务提交等，用完之后关闭SqlSession**。

1.1.1、优点

  1. 基于**SQL**语法，简单易学。
  2. 能了解底层组装过程。
  3. **SQL**语句封装在配置文件中，便于统一管理与维护，降低了程序的耦合度。
  4. 程序调试方便
  5. 与传统**JDBC**比较较少了大量的代码量，是最简单的持久化框架
  6. **SQL**代码从程序代码中彻底分离，可重用，增强了项目中的分工，增强了移植性

1.1.2、缺点

  1. **SQL**工作量很大，尤其是字段多、关联表多时，更是如此。
  2. **SQL**依赖于数据库，导致数据库移植性差。
  3. 框架还是比较简陋，功能尚有缺失，虽然简化了数据绑定代码，但是整个底层数据库查询实际还是要自己写的，工作量也比较大，而且不太容易适应快速数据库修改。
  4. 二级缓存机制不佳

1.2、SpringMVC核心知识点

​ SpringMVC是一款基于MVC思想的应用框架，它是Spring 的一个子框架。属于Spring 的一部分，SpringMVC的运行依赖Spring ，==两者之间无须第三方整合==。在MVC设计模式中，SpringMVC作为==控制器==(Controller)来建立模型与视图的数据交互。【直接点就是功能类似于**Servlet】。其核心就是使用前端控制器DispatcherServlet**。

1.2.1、执行思路

  1. 前端控制器**DispatcherServlet**接受客户端浏览器发送的请求。
  2. 前端控制器**DispatcherServlet调用处理器映射器HandlerMapping**查找相应的处理器。
  3. 处理器映射器根据**URL查找处理器Handler，并给前端控制器返回生成的处理器和相应的处理器拦截器HandlerIntercepter**。
  4. 前端控制器调用处理器适配器**HandlerAdapter**。
  5. 处理器适配器调用相应的处理器。
  6. 处理器给处理器适配器返回**ModelAndView**(包含逻辑视图名) 。
  7. 处理器将**ModelAndView**返回给前端控制器 。
  8. 前端控制器将**ModelAndView传给视图解析器ViewResolver **。
  9. 视图解析器**ViewResolver 解析后将具体的视图View**返回给前端控制器 。
  10. 前端控制器对视图进行渲染（视图渲染：将模型数据（在**ModelAndView中）填充到request**域中）。
  11. 前端控制器将视图返回给客户端浏览器。

1.2.2、Spring MVC的优势

  1. 清晰的角色划分：前端控制器（**DispatcherServlet），请求到处理器映射（HandlerMapping），处理器适配器（HandlerAdapter），视图解析器（ViewResolver ），处理器或页面控制器（Controller），验证器（Validator），命令对象（Command请求参数绑定到的对象就叫命令对象），表单对象（Form Object**提供给表单展示和提交到的对象就叫表单对象）。
  2. 分工明确，而且扩展点相当灵活，可以很容易扩展，虽然几乎不需要
  3. 由于命令对象就是一个**POJO，无需继承框架特定API**，可以使用命令对象直接作为业务对象
  4. ==和**Spring **框架无缝结合==，是其他web框架所不具备的
  5. 可适配，通过**HandlerAdapter**可以支持任意的类作为处理器
  6. 可定制行，**HandlerMapping，ViewResolver **等能够非常简单的定制
  7. 功能强大的数据验证，格式化，绑定机制；
  8. 利用**Spring 提供的Mock对象能够非常简单的进行WEB**层单元测试；
  9. 本地化，主题的解析的支持，使我们更容易进行国际化和主题的切换；
  10. 强大的**JSP标签库，使JSP**编写更容易。

1.2.3、SpringMVC常用注解：

@Controller
　　负责注册一个**bean 到Spring 上下文中。在Spring 中注册Controller**对象。

@RequestMapping
　　负责控制器指定可以处理哪些 **URL请求（设定请求方法访问URL**）

@RequestBody
　　该注解用于读取**request请求体中的部分数据，使用系统默认配置的HttpMessageConverter进行解析，然后把相应的数据绑定到要返回的对象上 ,再把HttpMessageConverter**返回的对象数据绑定到 **Controller中方法的参数上（请求数据转为JSON传递给Controller**中方法的参数上）

@ResponseBody
　　 该注解用于将**Controller的方法返回的对象，通过适当的HttpMessageConverter转换为指定格式后，写入到response对象的body数据区（响应数据对象转为JSON**）

1.3、Spring核心知识点

​ **Spring 是一个分层的、开源的轻量级框架。Spring**的核心有两部分：==控制反转、面向切面==。

1.3.1、IOC（Inversion on Control）意思是控制反转：

​ 举例来说，在之前的操作中，比方说有一个类，我们想要调用类里面的方法(不是静态方法)，就要创建类的对象，使用对象调用方法实现。对于**Spring来说，Spring创建对象的过程，不是在代码里面实现的，而是交给Spring**来进行配置实现的。

1.3.2、AOP（Aspect Oriented Programming）：面向切面编程。

​ 主要是管理==系统层的业务==，比如日志，权限，事物等。**AOP是将封装好的对象剖开，找出其中对多个对象产生影响的公共行为，并将其封装为一个可重用的模块，这个模块被命名为切面（aspect**），切面将那些与业务逻辑无关，却被业务模块共同调用的逻辑提取并封装起来，减少了系统中的重复代码，降低了模块间的耦合度，同时提高了系统的可维护性。

1.3.3、为什么使用Spring

  1. 方便解耦，简化开发。 **Spring就是一个大工厂，可以将所有对象的创建和依赖关系的维护，交给Spring**管理。
  2. **AOP编程的支持 。Spring**提供面向切面编程，可以方便的实现对程序进行权限拦截、运行监控等功能。
  3. 声明式事务的支持。只需要通过配置就可以完成对事务的管理，而无须手动编程。
  4. 方便程序的测试 。**Spring对Junit4支持，可以通过注解方便的测试Spring**程序。
  5. 方便集成各种优秀的框架 。**Spring不排斥各种优秀的开源框架，其内部提供了对各种优秀框架的直接支持(如：Struts2、Hibernate、MyBatis、Quartz**等)。
  6. 降低**JavaEE API的使用难度 。Spring对JavaEE开发中非常难用的一些API（JDBC、JavaMail、远程调用等），都提供了封装，使这些API**应用难度大大降低。

一、SSM整合思想

MyBatis+Spring+Spring MVC整合简称SSM

1、SSM整合步骤参考

  1. 搭建**SpringMVC环境。（控制器）==5.2.5**==
  2. 加入**MyBatis** 。（持久层）==3.4.6==
  3. 加入**Spring 。（依赖注入，事务——业务处理）==5.2.5**==
  4. 整合Spring与MyBatis以及优化业务层

^说明: SpringMVC 本身就是Spring中的一个子项目，所以==Spring与SpringMVC之间最大的优势在于无缝结合==，无需第三方整合。

2、分工说明

​ **SpringMVC：作为 View 层的实现者，完成用户的请求接收功能。SpringMVC中的Controller**作为整个应用的控制器，完成用户请求的转发及对用户的响应

​ MyBatis ：作为 **DAO**层的实现者，完成对数据库的增、删、改、查功能

​ **Spring **：以整个应用大管家的身份出现。整个应用中所有 Bean的生命周期行为，均由Spring 来管理。即整个应用中所有对象的创建、初始化、销毁，及对象间关联关系的维护，均由 **Spring **进行管理。

总结：

**SpringMVC**：负责业务流程控制（接收请求，跳转，响应等）。

MyBatis ：负责数据库交互。

**Spring **：负责整个项目中所有对象的初始化，以及业务处理（事务控制）。

		<servlet-name>dispatcherServlet</servlet-name>

	</servlet-mapping>

​ 在控制器（**Controller）类中，提供异常处理方法，在方法上使用注解@ExceptionHandler**，则此控制器中的任意方法中出现异常后，会自动调用该异常处理方法执行。

// @ExceptionHandler：不给异常类型时，处理所有异常
// 指定异常类型：处理指定异常（更加灵活，可以根据不同的异常跳转或显示不同页面）
@ExceptionHandler(value = NullPointerException.class)
public String exceptionMethod(Exception exception,Model model) {
  System.out.println("出错了，异常信息为："+exception.getMessage());
  System.out.println("出错了，异常类型为："+exception.getClass().getName());
  // 将异常信息发送给页面
  model.addAttribute("msg", exception.getMessage());
  return "error";// Spring MVC视图解析器处理调用到错误页面errot.jsp
}

@ControllerAdvice

[^异常处理器说明]: 自定义异常处理器，要求实现接口**HandlerExceptionResolver，实际在Spring3.0**及之后版本提供了默认的实现类 **SimpleMappingExceptionResolver**，如上一种异常处理配置便是。
